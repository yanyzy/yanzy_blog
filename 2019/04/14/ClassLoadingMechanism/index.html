JVM_类加载机制

Posted on 2019-04-14
虚拟机如何加载 Class 文件，加载后的 Class 文件又有什么变化

在 class 文件中描述的各种信息，都要加载到虚拟机之后才能运行。

类加载机制：虚拟机把描述类的数据从 class 文件加载到内存，并对数据进行校验，转换解析和初始化，最终形成可以被虚拟机直接使用的 java 类型。

类加载的时机

关于什么时候开始类加载的第一个阶段，虚拟机并没有强制规定，但是虚拟机规范严格规定了有且只有 5 种情况必须立即对类进行初始化（那么，加载，验证，准备自然需要在此之前开始）。

解析阶段可以在初始化之后进行

  • 遇到 new 、getstatic 、putstatic、invokestatic 这 4 条字节码指令时
  • 使用 java.lang.reflect 包的方法对类进行反射调用时
  • 当初始化一个类时，如果发现其父类没有被初始化，需要先对其父类进行初始化
  • 虚拟机启动时，用户需要指定一个要执行的主类，虚拟机会初始化这个主类
  • 使用 jdk 1.5 时，如果一个 MethodHandle 实例最后的解析结果句柄所对应的类没有初始化，则先触发其初始化

这 5 种行为称为对类的主动引用，除此之外的引用类的方式不会触发其初始化，称为被动引用

一个类在初始化的时候，要求其父类全部初始化过，但是一个接口在初始化时，并不要求其父接口全部完成初始化，只有在真正使用到父接口的时候才会初始化

类加载的过程

加载

加载阶段，虚拟机需要完成三件事：

  • 通过一个类的全限定名来获取一个类的二进制字节流
  • 将这个字节流所代表的的静态存储结构转化为方法区的运行时数据
  • 在内存中生成一个代表这个类的 Class 对象，作为方法区这个类的各种数据访问的入口

验证

验证是连接阶段的第一步，这一步是为了保证 Class 文件的字节流中包含的信息符合当前虚拟机的要求，并且不会危害虚拟机自身的安全。

验证大致会完成四个阶段的检验动作：

  • 文件格式验证：保证输入的字节流能正确地解析并存储于方法区中，格式符合要求。后面三个阶段的验证全部基于方法区的存储结构进行，不再操作字节流。
  • 元数据验证：对字节码描述的信息进行语义分析，保证其符合 java 语言规范的要求
  • 字节码验证：对类的方法体进行校验分析，保证被校验类的方法不会做出危害虚拟机安全的事件
  • 符号引用验证：确保解析动作能正常进行

准备

准备阶段是正式为类变量分配内存并设置初始值的阶段，这些变量所使用的内存都将在方法区分配（这里的初始值指的是 0 值，真正的赋值在初始化阶段）

解析

解析阶段是虚拟机将常量池里的符号引用替换为直接引用的过程。

初始化

初始化阶段，才真正执行类中定义的 java 程序代码。初始化阶段是执行类构造器方法的过程。

类加载器

类加载器：通过一个类的全限定名，获取定义这个类的二进制字节流。

类与类加载器

对于任意一个类，都需要由加载它的类加载器和这个类本身一同确立其在 java 虚拟机中的唯一性。即使两个类来自于用一个 class 文件，由同一虚拟机加载，只要加载它们的类加载器不同，那这两个类就必定不想等。

双亲委派模型

从 java 虚拟机的角度来讲，只存在两种不同的类加载器：

  • 启动类加载器 （Bootstrap ClassLoader)：使用 c++ 语言实现，是虚拟机自身的一部分
  • 其他所有类加载器：由 java 语言实现，独立于虚拟机外部，全部继承自 java.lang.ClassLoader

从开发人员的角度讲，大致可以分为三种类加载器：

  • 启动类加载器：加载 \lib 目录，并且被虚拟机识别的类库，无法被 java 程序直接使用。
  • 扩展类加载器：负责加载 \lib\ext 目录中的类库
  • 应用程序类加载器：也称为系统类加载器，负责加载用户路径指定的类库，一般是默认的类加载器

双亲委派模型：要求除启动类加载器以外，所有类加载器都有其父类加载器。其工作过程是：当一个类加载器收到了加载类的请求，首先自己不会尝试去加载这个类，而是把这个请求委派给父类加载器，每个类加载器都如此，因此所有加载请求都将最终传送到顶层的启动类加载器，只有当父类加载器反馈自己无法加载这个类，子加载器才尝试自己去加载。
